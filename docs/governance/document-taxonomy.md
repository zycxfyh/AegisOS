# Document Taxonomy

Status: **ACCEPTED** | Date: 2026-04-30 | Phase: DG-1
Tags: `governance`, `document`, `taxonomy`, `type-system`

## 1. Purpose

Every document in the Ordivon project carries a type. The type determines:
- What authority the document holds
- What freshness window is acceptable
- Who owns it (or should own it)
- Whether it can be archived
- What AI read priority it has

This taxonomy defines all document types and their properties.

## 2. Document Types

### 2.1 root_context

**Purpose**: Top-level entry point for AI agents and developers entering the project.

**Authority**: `source_of_truth` (for current project state and identity).

**Freshness expectation**: Must be updated within 1 session of any phase state change.
Stale threshold: 7 days.

**Owner expectation**: Project lead. Must be maintained during every phase transition.

**Archive rule**: Never archived while project is active. Previous versions replaced.

**AI read priority**: Level 0 — mandatory first read.

**Examples**: `AGENTS.md`, `CLAUDE.md`, `.cursorrules`

### 2.2 ai_onboarding

**Purpose**: AI-facing documentation that orients fresh agents on project identity,
architecture, boundaries, and working rules.

**Authority**: `current_status` (for AI behavioral constraints).

**Freshness expectation**: Must be updated within 1 session of any boundary change.
Stale threshold: 14 days.

**Owner expectation**: Project lead or delegated governance maintainer.

**Archive rule**: Never archived. Updated in place.

**AI read priority**: Level 1 — required for any agent performing work.

**Examples**: `docs/ai/README.md`, `docs/ai/current-phase-boundaries.md`,
`docs/ai/agent-working-rules.md`

### 2.3 phase_boundary

**Purpose**: Defines current active/inactive/deferred/NO-GO phase state.

**Authority**: `source_of_truth` (for phase boundaries).

**Freshness expectation**: Updated at every phase transition.
Stale threshold: 7 days.

**Owner expectation**: Project lead.

**Archive rule**: Never archived. Updated in place.

**AI read priority**: Level 1 — required before any phase-scoped work.

**Examples**: `docs/ai/current-phase-boundaries.md`

### 2.4 architecture

**Purpose**: Describes system structure, layers, import rules, and component relationships.

**Authority**: `current_status` (for structural decisions).

**Freshness expectation**: Updated when architecture changes.
Stale threshold: 30 days.

**Owner expectation**: Architecture lead or project lead.

**Archive rule**: Updated in place. Superseded versions may be archived.

**AI read priority**: Level 2 — required when making structural changes.

**Examples**: `docs/ai/architecture-file-map.md`, `docs/architecture/` files

### 2.5 design_spec

**Purpose**: Design system specs, component specs, token definitions, UI patterns.

**Authority**: `current_status` (for UI design, but not for implementation authorization).

**Freshness expectation**: Updated when design system changes.
Stale threshold: 30 days.

**Owner expectation**: Design lead.

**Archive rule**: Updated in place. Old specs archived.

**AI read priority**: Level 2 — required for UI work.

**Examples**: `docs/design/` files

### 2.6 runbook

**Purpose**: Operational procedures for running, deploying, or debugging the system.

**Authority**: `supporting_evidence` (operational guidance, not policy).

**Freshness expectation**: Updated when tooling/procedures change.
Stale threshold: 60 days.

**Owner expectation**: DevOps or project lead.

**Archive rule**: Updated in place. Superseded versions archived.

**AI read priority**: Level 3 — load when relevant.

**Examples**: `docs/runbooks/` files

### 2.7 receipt

**Purpose**: Immutable evidence that a specific action occurred — a trade was placed,
a decision was made, a review was completed.

**Authority**: `historical_record` (evidence, not current authority).

**Freshness expectation**: N/A — receipts are historical by nature. They do not stale.

**Owner expectation**: Generated by the system or phase. Not updated after creation.

**Archive rule**: Permanently retained. May be consolidated (e.g., 11→1 pattern from PT-001).

**AI read priority**: Level 4 — only when explicitly investigating that specific event.

**Examples**: PT-001.md, PT-002.md, cancel-pt004-001 receipt

### 2.8 stage_summit

**Purpose**: Formal phase closure document that seals a phase, records what was proved,
what was not proved, and establishes post-phase boundaries.

**Authority**: `current_status` (for what it closes and what it defers).

**Freshness expectation**: Published once at phase close. Not updated after.
Stale threshold: N/A — superseded, not stale.

**Owner expectation**: Project lead.

**Archive rule**: Never archived. Superseded by next phase's stage summit.

**AI read priority**: Level 2 — required for understanding phase transitions.

**Examples**: `docs/product/alpaca-paper-dogfood-stage-summit-phase-7p.md`

### 2.9 red_team

**Purpose**: Security review, attack surface analysis, failure mode catalog.

**Authority**: `supporting_evidence` (analysis, not policy).

**Freshness expectation**: Updated when security surface changes.
Stale threshold: 60 days.

**Owner expectation**: Security reviewer.

**Archive rule**: Retained. New red-team docs supersede old ones.

**AI read priority**: Level 3 — read when doing security-sensitive work.

**Examples**: `docs/runtime/policy-platform-red-team-closure.md`

### 2.10 ledger

**Purpose**: Machine-readable event evidence in JSONL format, checkable by invariant
verifiers.

**Authority**: `supporting_evidence` (evidence, NOT execution authority).

**Freshness expectation**: Updated per-event. Append-only. Individual events do not stale.

**Owner expectation**: System-generated. Checked by automated verifiers.

**Archive rule**: Permanently retained. Never deleted.

**AI read priority**: Level 3 — referenced by checkers, not primary reading.

**Examples**: `docs/runtime/paper-trades/paper-dogfood-ledger.jsonl`

### 2.11 tracker

**Purpose**: Tracks progress toward a future gate (e.g., Phase 8 readiness).

**Authority**: `current_status` (for gate criteria, not for authorization to proceed).

**Freshness expectation**: Updated as criteria change.
Stale threshold: 30 days.

**Owner expectation**: Project lead.

**Archive rule**: Archived when gate is met or deferred.

**AI read priority**: Level 3 — relevant when planning future phases.

**Examples**: `docs/runtime/paper-trades/phase-7p-readiness-tracker.md`

### 2.12 schema

**Purpose**: Defines data models, field specifications, invariants for machine-readable
evidence.

**Authority**: `source_of_truth` (for schema of its target format).

**Freshness expectation**: Updated when schema changes.
Stale threshold: 30 days.

**Owner expectation**: Domain owner.

**Archive rule**: Updated in place. Old versions archived.

**AI read priority**: Level 3 — required when writing to that format.

**Examples**: `docs/runtime/paper-trades/paper-dogfood-ledger-schema.md`

### 2.13 template

**Purpose**: Reusable structure for creating new documents of a specific type.

**Authority**: `example` (illustrative, not prescriptive).

**Freshness expectation**: Updated when format conventions change.
Stale threshold: 60 days.

**Owner expectation**: Governance maintainer.

**Archive rule**: Replaced when conventions change.

**AI read priority**: Level 4 — load when creating a new instance.

**Examples**: `docs/runtime/paper-trades/paper-trade-template.md`

### 2.14 adr

**Purpose**: Architecture Decision Record — documents a significant architectural
decision with context, options considered, and rationale.

**Authority**: `source_of_truth` (for the decision it records).

**Freshness expectation**: N/A once accepted. Superseded by later ADRs if reversed.
Stale threshold: N/A.

**Owner expectation**: Decision author.

**Archive rule**: Never archived. Superseded ADRs reference their replacement.

**AI read priority**: Level 2 — read when working in the affected subsystem.

### 2.15 archive_index

**Purpose**: Index into archived documents for historical lookup.

**Authority**: `archive` (navigation aid, not authority).

**Freshness expectation**: Updated when new archives are created.
Stale threshold: 90 days.

**Owner expectation**: Governance maintainer.

**Archive rule**: Updated in place.

**AI read priority**: Level 4 — only when deep historical context needed.

### 2.16 product

**Purpose**: Product strategy, roadmap, stage summits, pack contracts.

**Authority**: `current_status` (for product direction).

**Freshness expectation**: Updated when product decisions change.
Stale threshold: 30 days.

**Owner expectation**: Product lead.

**Archive rule**: Stage summits retained permanently. Roadmaps updated in place.

**AI read priority**: Level 2 — for understanding what to build.

**Examples**: `docs/product/` files

### 2.17 runtime

**Purpose**: Runtime evidence reports, dogfood protocols, operating plans, red-team closures.

**Authority**: `supporting_evidence` (most), `current_status` (active protocols).

**Freshness expectation**: Varies by subtype.
Stale threshold: 30-60 days depending on subtype.

**Owner expectation**: Domain owners.

**Archive rule**: Retained as evidence. Active protocols updated in place.

**AI read priority**: Level 3 — relevant for domain-specific work.

**Examples**: `docs/runtime/` files (excluding paper-trades ledger/tracker)

### 2.18 governance_pack

**Purpose**: Documents that define governance rules for a specific domain (this pack).

**Authority**: `current_status` (for governance rules, once accepted).

**Freshness expectation**: Updated when governance rules change.
Stale threshold: 30 days.

**Owner expectation**: Governance maintainer.

**Archive rule**: Updated in place. Old versions archived.

**AI read priority**: Level 2 — required for understanding document governance.

**Examples**: Files in `docs/governance/`

## 3. Priority Quick Reference

| Priority | Level | Types |
|----------|-------|-------|
| Mandatory (first read) | 0 | `root_context` |
| Required (before work) | 1 | `ai_onboarding`, `phase_boundary` |
| Important (before domain work) | 2 | `architecture`, `design_spec`, `stage_summit`, `adr`, `product`, `governance_pack` |
| Contextual (as needed) | 3 | `runbook`, `red_team`, `ledger`, `tracker`, `schema`, `runtime` |
| Reference (deep dive) | 4 | `receipt`, `template`, `archive_index` |

## 4. Authority Quick Reference

| Authority | For Decision-Making? | Types |
|-----------|---------------------|-------|
| `source_of_truth` | Yes | `root_context`, `phase_boundary`, `schema`, `adr` |
| `current_status` | Yes | `ai_onboarding`, `architecture`, `design_spec`, `stage_summit`, `tracker`, `product`, `governance_pack`, some `runtime` |
| `supporting_evidence` | Context only | `runbook`, `red_team`, `ledger`, most `runtime` |
| `historical_record` | No — evidence only | `receipt` |
| `proposal` | No — not yet accepted | Draft ADRs, proposed governance docs |
| `example` | No — illustrative | `template` |
| `archive` | No — historical interest | `archive_index` |
